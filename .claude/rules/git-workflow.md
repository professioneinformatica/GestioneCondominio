# Git Workflow

### Branch Strategy (Git Flow)

- **Development branch**: `develop` — main integration branch for all features
- **Main/Production branch**: `main`
- **Feature branches**: `feature/{feature-name}` — branched from and merged back into `develop`
- **Release branches**: `release/{version}` — branched from `develop` and merged back into `main` and `develop`
- **Bugfix branches**: `bugfix/{description}` — branched from `develop`
- **Hotfix branches**: `hotfix/{description}` — branched from `main`, merged into both `main` and `develop`

### Feature Branch Rule

When a requested change impacts **10 or more files**, ask the user before proceeding:
> "This change impacts N files. Do you want to create a new branch `feature/{name}`?"

Only proceed after explicit confirmation.

### Pre-Commit Secret Check

Before every commit, scan all staged files for hardcoded secrets, passwords, API keys, connection strings with credentials, tokens, or any other sensitive data. Look for patterns such as:

- Passwords or secrets in `appsettings.json`, `appsettings.*.json`, or any other configuration file
- Hardcoded connection strings containing `Password=`, `Pwd=`, or embedded credentials
- API keys, tokens, or client secrets assigned to variables or constants in code
- Private keys, certificates, or `.env` files

**If any secret or sensitive value is found, STOP the commit immediately** and report the finding to the user. Then suggest one of the following remediation strategies:

1. **Exclude the file from git** — add the file (or a pattern) to `.gitignore` if the file should never be tracked (e.g. `.env`, `appsettings.Development.json` with local secrets)
2. **Use a key vault or secret manager** — replace the hardcoded value with a reference to Azure Key Vault, .NET User Secrets (`dotnet user-secrets`), or environment variables, keeping the secret out of source control entirely

Do **not** proceed with the commit until the secret has been removed or the file has been excluded.

### Commit Message Format

Commits use an **extended Conventional Commits** format. The first line is the standard short header; the body contains a JSON payload with the full change metadata.

#### Header line

```
<type>(<scope>): <short description>
```

- When the commit contains a **single change**, the header `type` and `scope` match that change.
- When the commit contains **multiple changes**, the header `type` and `scope` are taken from the **first change** that has a description with `audience: "users"`. If no change has a `users` description, use the first change overall.
- The `<short description>` is derived from the `short` field of the selected change's `"users"` description. If no `users` description exists, use the `technical` one.
- Keep it under 100 characters.
- Written in **English**.

> **Note**: When possible, prefer one commit per logical change. Use multi-change commits only when changes are atomic and inseparable (e.g. a feature + the DB migration that supports it).

#### Body — JSON payload

The body must contain a single JSON object that conforms to the **Commit Schema** defined below. The JSON must be valid and pretty-printed. **A blank line is required between the header and the body** (standard git convention).

```
<type>(<scope>): <short description>

{
  "changes": [ ... ]
}
```

#### Non-conforming commits

Commits whose body does not contain a valid JSON payload conforming to the schema are **skipped** during changelog generation. They may be listed in a separate section for manual review. This includes merge commits, automated tool commits, and any manually written commit without a JSON body.

### Commit Schema

```json
{
  "$schema": "https://json-schema.org/draft/2020-12",
  "type": "object",
  "required": ["changes"],
  "properties": {
    "changes": {
      "type": "array",
      "minItems": 1,
      "items": { "$ref": "#/$defs/change" }
    }
  },
  "$defs": {
    "change": {
      "type": "object",
      "required": ["type", "scope", "descriptions"],
      "properties": {
        "type": {
          "type": "string",
          "enum": ["feat", "fix", "refactor", "perf", "chore", "docs", "test", "style", "ci", "build"]
        },
        "scope": {
          "type": "string",
          "description": "Project area (e.g. condominiums, accounting, assembly)"
        },
        "descriptions": {
          "type": "array",
          "minItems": 1,
          "maxItems": 2,
          "items": { "$ref": "#/$defs/description" }
        }
      }
    },
    "description": {
      "type": "object",
      "required": ["audience", "short"],
      "properties": {
        "audience": {
          "type": "string",
          "enum": ["users", "technical"],
          "description": "Who this description is intended for"
        },
        "breaking": {
          "type": "boolean",
          "default": false,
          "description": "Whether this is a breaking change for this specific audience"
        },
        "short": {
          "type": "string",
          "maxLength": 100,
          "description": "One-line summary in English. Used in the ChangeLog (users) and git log (technical)."
        },
        "body": {
          "type": "string",
          "description": "Optional extended description in English. May be multi-line."
        }
      }
    }
  }
}
```

> **Note on `descriptions` uniqueness**: the schema allows at most 2 entries. The rule "at most one description per `audience`" is enforced by convention: do not include two descriptions with the same `audience` value in a single change.

#### Schema rules

| Field | Level | Required | Notes |
|-------|-------|----------|-------|
| `changes` | root | yes | Array of at least 1 change |
| `type` | `change` | yes | Conventional Commit type |
| `scope` | `change` | yes | Project scope (see table below) |
| `descriptions` | `change` | yes | 1 or 2 entries, at most one per audience |
| `audience` | `description` | yes | `"users"` or `"technical"` |
| `breaking` | `description` | no | Default `false`. Breaking is audience-specific: a UI redesign can be breaking for users but not for developers; an internal API change can be breaking for developers but transparent to users. |
| `short` | `description` | yes | Max 100 characters, always in English |
| `body` | `description` | no | Free text in English, supports newlines (`\n`) |

#### Audience semantics

- `"users"` — change visible to or relevant for end users → **included in the ChangeLog**
- `"technical"` — internal change for developers, sysadmins, CI/CD → **excluded from the ChangeLog**

A `change` must have at least one `description`. Having both (`users` + `technical`) is optional but recommended when the change is relevant to both audiences.

### Examples

#### Bug fix with both audiences

```
fix(accounting): fix expense distribution when apportionment table sums to less than 1000

{
  "changes": [
    {
      "type": "fix",
      "scope": "accounting",
      "descriptions": [
        {
          "audience": "users",
          "short": "Fixed incorrect expense distribution with incomplete apportionment tables",
          "body": "When an apportionment table had a total less than 1000 (e.g. due to vacant units), expenses were distributed incorrectly. The system now normalizes the distribution based on actual totals."
        },
        {
          "audience": "technical",
          "short": "Normalize distribution ratios when apportionment total != 1000",
          "body": "ExpenseDistributionService.DistributeAsync now divides each unit's millesimal value by the actual sum instead of assuming 1000. Added guard clause for zero-sum tables."
        }
      ]
    }
  ]
}
```

#### Feature touching multiple areas

The header uses the `type` and `scope` of the first change that has a `users` description.

```
feat(assembly): add online voting support for assembly resolutions

{
  "changes": [
    {
      "type": "feat",
      "scope": "assembly",
      "descriptions": [
        {
          "audience": "users",
          "short": "Added online voting support for assembly resolutions"
        }
      ]
    },
    {
      "type": "chore",
      "scope": "db",
      "descriptions": [
        {
          "audience": "technical",
          "short": "Add migration for OnlineVote and VotingSession tables"
        }
      ]
    }
  ]
}
```

#### Breaking change — different impact per audience

```
refactor(owners): redesign property owner portal

{
  "changes": [
    {
      "type": "refactor",
      "scope": "owners",
      "descriptions": [
        {
          "audience": "users",
          "breaking": true,
          "short": "Property owner portal has been fully redesigned",
          "body": "The portal layout and navigation have changed significantly. Previous bookmarks may no longer work."
        },
        {
          "audience": "technical",
          "breaking": false,
          "short": "Replaced owner portal with MudBlazor dashboard layout"
        }
      ]
    }
  ]
}
```

### Version Format

`{year}.{quarter}.{MMdd}.{HHmm}` — Example: `2026.1.0308.1430`

| Segment | Description | Example |
|---------|-------------|---------|
| `year` | 4-digit year | `2026` |
| `quarter` | Quarter number 1–4 | `1` |
| `MMdd` | 2-digit month + 2-digit day (both zero-padded) | Mar 08 → `0308`, Oct 15 → `1015` |
| `HHmm` | 24-hour hour and minutes | `1430` |

The `quarter` is derived from the month: 01–03 = Q1, 04–06 = Q2, 07–09 = Q3, 10–12 = Q4. Ensure consistency between `quarter` and `MMdd`.

Git tags use the same format (e.g., `git tag 2026.1.0308.1430`).

### Changelog

Changelog files are located in `ChangeLog/` at the solution root.

- **`ChangeLog/ChangeLog-{version}.md`**: Immutable archived changelog for a specific released version.

Changelog files are always **bilingual (Italian and English)**, with a fully separate section for each language:

- Only descriptions with `"audience": "users"` are included in the ChangeLog.
- Entries are **grouped by area (scope)** within each language section.
- **Scope sections are sorted alphabetically.**
- Each entry uses an **emoji** to indicate the commit type (see legend below).
- Breaking changes are marked with **⚠️** after the emoji type.
- On the same line of the emoji write the `short` text, followed by ` — ` and the **commit date** (`YYYY-MM-DD` from `git log --format="%as"`). If the `body` is available, add it on indented continuation lines.
- Sort the changes for every area by commit type: `feat` → `fix` → `refactor` → `perf` → `chore` → `docs` → `test` → `style` → `ci` → `build`.
- Use an unordered list markdown.

> **Note**: Types such as `ci`, `build`, `test`, and `style` appear in the changelog only when the change has an explicit `users` description, which is rare.

**Emoji legend by commit type:**

| Emoji | Type | Meaning |
|-------|------|---------|
| ✨ | `feat` | New feature |
| 🐛 | `fix` | Bug fix |
| ♻️ | `refactor` | Refactoring |
| ⚡️ | `perf` | Performance improvement |
| 🔧 | `chore` | Maintenance / tooling |
| 📝 | `docs` | Documentation |
| 🧪 | `test` | Tests |
| 🎨 | `style` | Code style / formatting |
| 👷 | `ci` | CI/CD pipeline |
| 📦 | `build` | Build system |

**Format example** (Italian text is translated from English by the AI agent using the domain glossary):

```markdown
---

## 🇮🇹 Italiano

### Assemblee
- ✨ Aggiunto supporto voto online per le delibere assembleari — 2026-03-08

### Contabilita
- 🐛 Corretta ripartizione spese con tabelle millesimali incomplete — 2026-03-08
  Quando una tabella millesimale aveva un totale inferiore a 1000 (es. per unita vacanti),
  le spese venivano ripartite in modo errato.

---

## 🇬🇧 English

### Assemblies
- ✨ Added online voting support for assembly resolutions — 2026-03-08

### Accounting
- 🐛 Fixed incorrect expense distribution with incomplete apportionment tables — 2026-03-08
  When an apportionment table had a total less than 1000 (e.g. due to vacant units),
  expenses were distributed incorrectly.
```

### Release Process

When asked to prepare a release, follow these steps **in order**:

> **Error handling**: if any git command fails (merge conflict, push rejected, tag already exists, etc.), **stop immediately** and report the error to the user. Do not proceed to the next step. The release process must be completed in full or not at all — partial releases leave the repository in an inconsistent state.

#### 1. Determine the version number

Compute the version using the format `{year}.{quarter}.{MMdd}.{HHmm}` based on the current date and time.

#### 2. Create the release branch

```bash
git checkout develop
git pull origin develop
git checkout -b release/{version}
```

#### 3. Collect commits to release

Identify all commits since the last release (i.e., since the last merge into `main`):

```bash
git log main..HEAD --format="%H %as" --reverse
```

For each commit, extract the JSON payload from the commit body:

```bash
git show <hash> --format="%B" -s
```

Parse the JSON and collect all `change` entries whose `descriptions` array contains at least one entry with `"audience": "users"`. Skip commits whose body does not contain a valid JSON payload (merge commits, non-conforming commits, etc.).

#### 4. Generate `ChangeLog/ChangeLog-{version}.md`

Build the changelog file from the collected entries following the format described in the Changelog section above. Translate Italian entries using the domain glossary.

#### 5. Update the version in `Directory.Build.props`

Update the three version properties in `Directory.Build.props` at the solution root:

```xml
<AssemblyVersion>{version}</AssemblyVersion>
<FileVersion>{version}</FileVersion>
<Version>{version}</Version>
```

#### 6. Commit on the release branch

Stage and commit both the new changelog file and the updated `Directory.Build.props`:

```
chore(config): release {version}

{
  "changes": [
    {
      "type": "chore",
      "scope": "config",
      "descriptions": [
        {
          "audience": "technical",
          "short": "Release {version} — update version and generate changelog"
        }
      ]
    }
  ]
}
```

This release commit itself is **not** included in the changelog it generates.

#### 7. Merge to `main`, tag, merge back to `develop`, and push

```bash
# Merge into main
git checkout main
git merge --no-ff release/{version}
git tag {version}

# Merge back into develop
git checkout develop
git merge --no-ff release/{version}

# Delete the release branch
git branch -d release/{version}

# Push everything
git push origin main
git push origin develop
git push origin {version}
git push origin --delete release/{version}
```
